\section{Arbeitstitel}\label{sec:introduction}

Vergleich einer bestehenden Single-Agent-Review-Lösung mit einer erweiterten Multi-Agent-Architektur zur automatisierten Qualitätskontrolle in CI-Pipelines



\subsection{Problemstellung und Motivation}

In modernen Softwareentwicklungsprozessen ist die Qualitätssicherung ein zentraler Bestandteil des Entwicklungszyklus. Continuous Integration (CI) und Merge-Request-basierte Workflows, insbesondere auf Plattformen wie \textit{GitLab}, ermöglichen schnelle Iterationen, erhöhen jedoch gleichzeitig den Bedarf an effizienter und skalierbarer Qualitätssicherung.

Im betrachteten Unternehmenskontext existiert bereits ein produktiv eingesetzter KI-basierter Single-Agent. Dieser analysiert Git-Diffs innerhalb von Merge Requests und generiert textuelle Review-Kommentare unter Berücksichtigung projektspezifischer Standards und typischer Fehlerquellen. 

Die bestehende Lösung weist jedoch funktionale Einschränkungen auf:

\begin{itemize}
	\item Es erfolgt keine funktionale Trennung verschiedener Qualitätsdimensionen.
	\item Der Agent kann ausschließlich Kommentare in bestehenden Merge Requests hinterlassen.
	\item Es werden keine separaten Review-Branches erzeugt.
	\item Es erfolgt keine eigenständige Nutzung externer APIs oder automatisierte Codeänderung.
\end{itemize}

Die aktuelle Architektur bildet somit eine kommentierende Assistenz, jedoch keine aktive, strukturierte Review-Instanz mit automatisierter Änderungsintegration.

Mit dem Aufkommen leistungsfähiger Large Language Models und agentenbasierter Orchestrierungskonzepte stellt sich die Frage, ob eine funktional spezialisierte Multi-Agent-Architektur einen messbaren Mehrwert gegenüber diesem bestehenden Single-Agent-Ansatz liefern kann.

Insbesondere ist offen:

\begin{itemize}
	\item Ob funktionale Spezialisierung zu höherer Präzision oder geringerer Fehlerrate führt,
	\item Ob automatisierte Branch-Erstellung und Merge-Request-Generierung die Prozessstabilität erhöhen oder senken,
	\item Ob sich die Akzeptanz bei Entwicklerinnen und Entwicklern verändert,
	\item Wie sich Kosten, Laufzeit und Ressourcenverbrauch entwickeln.
\end{itemize}

Vor diesem Hintergrund zielt die Arbeit nicht auf die erstmalige Einführung eines KI-Review-Systems, sondern auf die systematische Weiterentwicklung und den empirischen Vergleich zweier Architekturen im realen Unternehmensumfeld.



\subsection{Zielsetzung der Arbeit}

Ziel dieser Arbeit ist der systematische Vergleich eines bestehenden Single-Agent-Review-Systems mit einer neu konzipierten Multi-Agent-Architektur, die funktional erweiterte Fähigkeiten besitzt.

Hierzu wird:

\begin{itemize}
	\item eine Multi-Agent-Architektur konzipiert,
	\item technisch implementiert,
	\item in die bestehende CI-Pipeline integriert,
	\item und unter realen Bedingungen mit dem bestehenden Single-Agent verglichen.
\end{itemize}

Die neu entwickelte Multi-Agent-Architektur soll zusätzlich zu kommentarbasierten Reviews folgende neue Funktionen bereitstellen:

\begin{itemize}
	\item funktionale Spezialisierung einzelner Sub-Agenten,
	\item Orchestrierung durch einen Master-Agent,
	\item automatisierte Erstellung dedizierter Review-Branches,
	\item eigenständige Codeanpassungen,
	\item automatisierte Erstellung von Merge Requests,
	\item kontrollierte Nutzung externer APIs,
	\item Implementierung technischer Rekursionsschutzmechanismen.
\end{itemize}

Ziel ist es zu untersuchen, ob die erweiterte Multi-Agent-Architektur im Vergleich zum bestehenden Single-Agent:

\begin{itemize}
	\item eine höhere Qualität der Verbesserungsvorschläge liefert,
	\item eine geringere Fehlerrate (False Positives) aufweist,
	\item eine höhere Akzeptanzrate generierter Änderungen erzielt,
	\item messbare Verbesserungen hinsichtlich Testabdeckung oder Codequalität bewirkt,
    \item eine verringerung des Auwandes für Entwickler und Entwicklerinnen bittet,
	\item und ein vertretbares Kosten-Nutzen-Verhältnis aufweist.
\end{itemize}



\subsection{Forschungsfrage}

\subsubsection{Zentrale Forschungsfrage}

Führt eine funktional spezialisierte Multi-Agent-Architektur mit erweiterten Automatisierungsfähigkeiten im Vergleich zu einem bestehenden Single-Agent-Review-System zu einer messbaren Verbesserung der Codequalität und Prozessstabilität in CI-basierten Merge-Request-Workflows?

\subsubsection{Teilforschungsfragen}

\begin{itemize}
	\item Unterscheidet sich die Qualität der generierten Verbesserungsvorschläge signifikant zwischen Single- und Multi-Agent?
	\item Wie verändert sich die Akzeptanzrate AI-generierter Änderungen?
	\item Welche Auswirkungen hat die automatisierte Branch- und Merge-Request-Erstellung auf den Entwicklungsprozess?
	\item Wie verändern sich Tokenverbrauch, Laufzeit und Infrastrukturkosten?
	\item Welche Governance- und Rekursionsmechanismen sind für eine sichere Integration erforderlich?
\end{itemize}



\subsection{Stand der Technik}

Die Arbeit stützt sich auf folgende technologische und konzeptionelle Grundlagen:

\begin{itemize}
	\item Continuous Integration und Merge-Request-Workflows auf \textit{GitLab},
	\item Large Language Models (z. B. über \textit{AWS Bedrock}),
	\item automatisierte Code-Analyse- und Coverage-Tools wie \textit{JaCoCo},
	\item agentenbasierte Orchestrierungskonzepte,
	\item Human-AI-Collaboration im Software Engineering.
\end{itemize}

Darüber hinaus werden regulatorische Rahmenbedingungen, insbesondere der \textit{EU AI Act}, im Hinblick auf Governance und Nachvollziehbarkeit berücksichtigt.



\section{Konzeption der Vergleichsarchitekturen}

Dieses Kapitel definiert die zu untersuchenden Architekturen und schafft die Grundlage für eine systematische Evaluation.



\subsection{Bestehender Single-Agent (Baseline)}

Der bestehende Single-Agent analysiert:

\begin{itemize}
	\item Git-Diffs von Merge Requests,
	\item projektspezifische Standards,
	\item typische Fehlerquellen.
\end{itemize}

Er erzeugt ausschließlich textuelle Kommentare innerhalb der bestehenden Merge Request.

Charakteristika:

\begin{itemize}
	\item monolithische Entscheidungslogik,
	\item keine funktionale Spezialisierung,
	\item keine automatisierte Codeänderung,
	\item keine Branch-Erstellung,
	\item keine eigenständige API-Integration.
\end{itemize}

Dieser Agent dient als empirische Baseline.



\subsection{Neu zu implementierende Multi-Agent-Architektur}

Die Multi-Agent-Architektur verfolgt einen modularen, funktional spezialisierten Ansatz.

Ein Master-Agent übernimmt die Orchestrierung und aktiviert kontextabhängig spezialisierte Sub-Agenten.

\subsubsection{Master-Agent}

Aufgaben:

\begin{itemize}
	\item Analyse und Klassifikation des Git-Diffs,
	\item Entscheidung über Aktivierung relevanter Sub-Agenten,
	\item Erstellung eines dedizierten Review-Branches,
	\item Aggregation der Ergebnisse,
	\item Steuerung der automatisierten Merge-Request-Erstellung,
	\item Implementierung von Rekursionsschutzmechanismen.
\end{itemize}

\subsubsection{Spezialisierte Sub-Agenten}

Beispielsweise:

\begin{itemize}
	\item Coding-Standards-Agent,
	\item Korrektheits-Agent,
	\item Testabdeckungs-Agent,
	\item Fachlichkeits-Agent.
\end{itemize}

Diese können eigenständig Codeanpassungen vorschlagen oder implementieren.

\section{Evaluationsdesign}

Zur vergleichenden Analyse wird ein kontrolliertes Untersuchungsdesign gewählt, in dem beide Architekturen unter identischen Rahmenbedingungen betrieben werden.

Messgrößen umfassen:

\begin{itemize}
	\item Anzahl und Art generierter Verbesserungsvorschläge,
	\item False-Positive-Rate,
	\item Akzeptanzrate generierter Änderungen,
	\item Veränderung der Testabdeckung,
	\item Laufzeit pro Merge Request,
	\item Tokenverbrauch und Kostenstruktur.
\end{itemize}

Zusätzlich werden qualitative Entwicklerbefragungen durchgeführt, um Vertrauen, Transparenz und wahrgenommene Nützlichkeit zu erfassen.
